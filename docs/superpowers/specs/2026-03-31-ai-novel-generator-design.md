# AI 小说生成器设计文档

## 1. 目标

构建一个内部使用的 AI 小说生产后台，支持以下能力：

- 根据用户输入的大概内容、小说类型、预期章节数生成长篇小说
- 使用多 agent 分工协作，包括大纲、分部、章节、审核、改写
- 支持关键剧情节点的人机对话确认
- 支持每天自动生成固定数量章节
- 支持多本小说并行生成
- 支持多平台上传发布
- 支持多 provider、多 model、多 key 的统一调度
- 对不同 `provider + model + key` 组合施加独立并发和速率限制
- 使用 `superpower` 作为流程编排核心
- 提供完整前端后台页面
- 提供每个 agent 的提示词配置页面
- 不做登录与权限系统

## 2. 范围

本设计面向单管理员的内部生产后台，不包含：

- 多租户
- 用户注册登录
- 团队协作权限
- 对外计费
- 公共开放 API

系统目标是提高长篇小说批量生产的稳定性、可控性和可观测性，而不是追求 agent 的高度自治。

## 3. 设计原则

1. 流程控制优先于 agent 自治
2. 故事状态必须结构化持久化，不能依赖上下文记忆
3. 模型调用必须统一经过调度层
4. 人工只介入高价值剧情节点
5. 发布链路与生成链路解耦
6. 前端是生产控制台，不是展示站点

## 4. 总体架构

系统拆分为 6 个核心模块：

### 4.1 项目中心

负责小说项目的创建和配置，包括：

- 题材
- 简介
- 风格要求
- 目标章节数
- 每日生成章节数
- 目标发布平台
- 当前进度和状态

### 4.2 故事状态中心

负责维护所有 agent 共用的故事真相源，包括：

- 世界观
- 人物卡
- 总纲
- 分部计划
- 单章计划
- 已生成章节摘要
- 已确认关键剧情事实
- 禁忌规则

所有写作相关流程都从这里读取状态，并向这里写入结果。

### 4.3 Agent 执行层

负责执行具体写作与审核任务，包括：

- 大纲 agent
- 分部 agent
- 章节计划 agent
- 章节正文 agent
- 审核 agent
- 改写 agent
- 剧情确认助手

这些 agent 不直接驱动系统主流程。它们只接收输入，产出结构化输出。

### 4.4 工作流编排层

使用 `superpower` 定义和管理业务流程：

- 建项目
- 生成大纲
- 生成分部
- 生成章节计划
- 生成章节正文
- 审核和改写
- 关键节点人工确认
- 发布
- 定时生产

### 4.5 LLM 调度层

统一管理所有大模型调用。每次调用都必须先通过调度器分配可用的 `provider + model + key` 容量。

### 4.6 发布中心

负责对接固定平台连接器，完成：

- 平台配置校验
- 章节内容转换
- 上传发布
- 发布状态记录
- 失败重试

## 5. 子系统边界

### 5.1 内容生产核心

负责故事生成与一致性维护。包含：

- 故事状态中心
- agent 执行层
- 写作工作流

### 5.2 LLM 调度核心

负责 provider、model、key 的分配与限制。包含：

- key 池
- 并发槽位控制
- RPM 和 TPM 限制
- 预算限制
- fallback 路由

### 5.3 发布与平台集成

负责固定平台连接器和发布链路，不参与内容生成决策。

### 5.4 内部控制台

负责配置、监控和人工决策，不承载任何核心业务逻辑。

## 6. 核心工作流

### 6.1 CreateProjectFlow

输入：

- 小说大概内容
- 小说类型
- 预期章节数
- 风格要求
- 每天生成章节数
- 目标平台

输出：

- `NovelProject`
- 初始故事上下文

### 6.2 GenerateOutlineFlow

步骤：

1. 大纲 agent 生成总纲
2. 审核 agent 做结构审核
3. 通过后写入故事状态中心

输出内容包括：

- 世界观基础定义
- 主线冲突
- 核心角色集合
- 长线剧情规划
- 预期结局

### 6.3 GenerateVolumePlanFlow

步骤：

1. 分部 agent 按目标章节数拆分卷/部
2. 审核 agent 检查结构平衡
3. 写入 `VolumePlan`

每一部包含：

- 目标章节范围
- 剧情任务
- 角色推进目标
- 伏笔埋设与回收目标

### 6.4 GenerateChapterPlanFlow

正文生成前先生成章节计划，避免正文直接失控。

章节计划包含：

- 本章目标
- 关键情节节拍
- 视角人物
- 冲突推进点
- 本章要延续或回收的伏笔
- 不可违反的故事事实

### 6.5 GenerateChapterDraftFlow

步骤：

1. 读取故事状态
2. 获取 LLM 容量租约
3. 生成章节正文
4. 写入草稿

章节正文生成时只注入有限上下文：

- 故事 bible 摘要
- 当前卷计划
- 最近若干章摘要
- 当前章节计划

不把整本小说正文反复塞入上下文。

### 6.6 ReviewAndReviseFlow

步骤：

1. 审核 agent 对草稿做结构化审查
2. 判断通过、返修或触发关键节点确认
3. 普通问题进入改写 agent
4. 改写完成后再次审核
5. 通过后进入下一阶段

审核项包括：

- 世界观和设定冲突
- 人物行为偏移
- 节奏失衡
- 风格漂移
- 与章节计划不一致
- 是否命中关键剧情节点

### 6.7 DecisionSessionFlow

当章节命中关键剧情节点时，流程暂停，进入人工确认页。

系统先生成一个剧情决策包：

- 当前剧情背景
- 最近章节摘要
- 当前关键节点提案
- 风险点
- 可选方向

然后由管理员与剧情确认助手进行对话。

管理员最终给出结构化决议：

- 采用当前方案
- 采用某个替代方案
- 重新规划该节点
- 暂停后续章节推进

系统将决议写回故事状态中心，后续章节计划必须继承该决议。

### 6.8 PublishFlow

步骤：

1. 检查章节是否通过审核并完成关键节点确认
2. 生成平台 payload
3. 调用平台连接器上传
4. 保存发布状态和外部 ID
5. 失败时进入重试队列

### 6.9 ScheduledGenerationFlow

定时检查所有活跃小说项目：

- 是否在生成窗口内
- 当日已生成多少章
- 是否低于每日目标
- 是否被关键节点确认阻塞
- 是否因预算或限流暂停

满足条件则补齐当日生成任务。

## 7. Agent 设计

### 7.1 大纲 agent

职责：

- 根据题材、简介、章节数生成全书骨架

不负责：

- 分卷
- 分章
- 正文写作

输出：

- 结构化总纲
- 世界观概要
- 核心人物集
- 结局方向

### 7.2 分部 agent

职责：

- 将总纲拆分为多个卷/部

输出：

- 每卷目标
- 每卷章节范围
- 角色弧线推进目标

### 7.3 章节计划 agent

职责：

- 根据当前卷目标与历史状态生成单章计划

输出：

- 单章结构化计划

### 7.4 章节正文 agent

职责：

- 将单章计划扩写为正文

输出：

- 章节标题
- 章节摘要
- 正文草稿

### 7.5 审核 agent

职责：

- 发现问题并给出结构化修改意见

不直接负责正文改写。

### 7.6 改写 agent

职责：

- 按审核意见定向修订草稿

### 7.7 剧情确认助手

职责：

- 围绕关键剧情节点与管理员对话
- 提供替代走向和风险分析

限制：

- 只能建议
- 不能直接写入最终故事状态

## 8. 人工确认页设计

人工确认页不是简单审批页面，而是剧情决策对话台。

页面包含三个区域：

### 8.1 节点摘要区

展示：

- 为什么触发关键节点确认
- 当前剧情位置
- 涉及角色
- 影响的伏笔和后续章节

### 8.2 剧情对话区

支持管理员与剧情确认助手围绕当前节点对话，例如：

- 当前转折是否过早
- 有哪些更稳妥的推进路线
- 若保留当前方向，后续三章如何承接

### 8.3 决议面板

将最终结论固化为结构化指令，例如：

- 角色 A 不在本章黑化
- 改为关系裂痕伏笔
- 影响第 28 到 31 章计划

最终只有结构化决议对象能写回系统。

## 9. LLM 调度设计

### 9.1 调度单位

系统按 `provider + model + key` 作为最小容量单位。

每个容量桶包含：

- `maxConcurrentRequests`
- `requestsPerMinute`
- `tokensPerMinute`
- `dailyBudget`
- `enabled`
- `priority`
- `fallbackOrder`

### 9.2 调用流程

每个 agent 请求模型时必须声明：

- agent 类型
- 首选 provider/model
- fallback 列表
- 预计输入 token
- 预计输出 token
- 优先级

调度器执行：

1. 选择可用容量桶
2. 发放容量租约
3. 记录执行前预算占用
4. 发起模型调用
5. 记录真实 token 和成本
6. 释放租约

### 9.3 调度目标

- 避免并发超限
- 避免单 key 被打爆
- 支持模型分层
- 支持 provider 降级和故障转移
- 支持预算控制

### 9.4 模型分层建议

- 大纲和审核使用较强模型
- 章节计划使用中等模型
- 正文生成使用性价比模型
- 剧情确认助手使用较强模型

## 10. 并行策略

### 10.1 跨小说并行

允许多本小说并行生成。

### 10.2 单小说并行限制

首版强约束：

- 同一本小说同一时刻只允许一个正文生成任务

理由：

- 降低长篇一致性漂移
- 简化故事状态更新
- 降低关键节点冲突

### 10.3 可并行的辅助任务

允许与正文主链路并行的任务：

- 下一章计划预生成
- 发布任务
- 数据索引与摘要更新
- 非阻塞型统计任务

提示词不能硬编码在代码中。每个 agent 的提示词配置包含：

- `systemPrompt`
- `taskTemplate`
- `outputSchema`
- `reviewRubric`
- `version`
- `enabled`
- `lastTestedModel`

要求：

- 支持版本化
- 支持回滚
- 支持渲染预览
- 支持测试运行
- 支持按 agent 独立生效

## 13. 上下文装配规范

不同 agent 的上下文必须不一致，并且由系统统一装配，不能由 agent 自由决定拉取范围。

### 13.1 上下文来源分层

系统按以下四层装配上下文：

- `固定上下文`
  - 小说基础信息
  - 风格要求
  - 平台限制
  - 通用禁忌
- `故事状态上下文`
  - 世界观摘要
  - 人物卡摘要
  - 已确认总纲
  - 已确认分部计划
  - 已确认关键事实
  - 未回收伏笔
- `近期运行上下文`
  - 当前卷摘要
  - 最近若干章摘要
  - 上一章结果
  - 当前待承接剧情点
- `任务上下文`
  - 当前任务目标
  - 输入材料
  - 输出 schema
  - 质量要求

### 13.2 装配原则

1. 正文类 agent 默认只读取摘要层，不读取全书全文
2. 审核类 agent 必须同时看到草稿、计划和已确认事实
3. 改写类 agent 只拿完成修订所需的最小上下文
4. 关键节点确认助手只围绕当前节点取上下文
5. 上下文裁剪由系统完成，不允许 agent 自主扩展

### 13.3 Agent 上下文矩阵

#### 大纲 agent

必选：

- 用户原始创意
- 小说类型
- 目标章节数
- 风格要求
- 平台限制

可选：

- 已有参考作品标签
- 用户额外偏好

不应包含：

- 章节正文
- 分章历史

#### 分部 agent

必选：

- 已确认总纲
- 目标章节数
- 节奏要求
- 关键主线
- 预期结局方向

可选：

- 平台篇幅偏好

不应包含：

- 章节正文草稿

#### 章节计划 agent

必选：

- 世界观摘要
- 人物卡摘要
- 当前卷计划
- 最近若干章摘要
- 未回收伏笔
- 当前章序号
- 当前章目标

可选：

- 上一章正文末尾片段

不应包含：

- 全书正文

#### 章节正文 agent

必选：

- 当前章节计划
- 当前卷摘要
- 最近若干章摘要
- 硬性事实约束
- 风格约束

可选：

- 上一章结尾片段
- 关键人物口吻约束

不应包含：

- 全量总纲原文
- 全书正文

#### 审核 agent

必选：

- 当前章节草稿
- 当前章节计划
- 世界观摘要
- 人物状态摘要
- 最近若干章摘要
- 审核 rubric

可选：

- 上一版草稿

不应省略：

- 章节计划
- 已确认事实

#### 改写 agent

必选：

- 当前草稿
- 审核问题清单
- 允许修改范围
- 不可改动约束

可选：

- 相关设定摘要

不应包含：

- 与本次问题无关的大量历史内容

#### 剧情确认助手

必选：

- 当前关键节点提案
- 最近剧情摘要
- 当前卷目标
- 候选替代走向
- 风险分析规则

可选：

- 涉及角色的长期弧线摘要

不应包含：

- 无关章节正文

### 13.4 token 预算原则

上下文应设置预算上限，并按优先级裁剪：

1. 任务上下文
2. 已确认关键事实
3. 当前卷摘要
4. 最近章节摘要
5. 可选补充材料

裁剪时优先删除低价值的历史性描述，不删除硬性事实和当前任务要求。

### 13.5 摘要策略

系统应维护多层摘要，而不是只保留原文：

- 单章摘要
- 当前卷摘要
- 人物状态摘要
- 伏笔状态摘要
- 已确认事实摘要

正文 agent 和审核 agent 默认消费这些摘要层。只有进入特殊修复流程时，系统才允许附加原文片段。

## 15. 技术方案

推荐技术栈：

- 前端：`Next.js`
- 后端 API：`Node.js + TypeScript`

## 16. 错误处理

### 16.1 生成错误

- 模型调用失败时重试
- 不可恢复错误进入死信记录
- 超预算时暂停对应项目或降级模型

### 16.2 审核错误

- 自动改写最多有限次数
- 超过次数后挂起等待人工处理

### 16.3 发布错误

- 发布失败不回滚已通过审核的章节
- 进入单独重试队列

### 16.4 决策阻塞

- 关键节点未确认时暂停该小说的后续正文生成
- 不影响其他小说继续运行

## 17. 测试策略

### 17.1 单元测试

- 故事状态更新逻辑
- 调度器选桶逻辑
- 限流逻辑
- 平台 payload 转换

### 17.2 集成测试

- 建项目到出大纲
- 章节生成到审核通过
- 关键节点阻塞到恢复
- 发布流程

### 17.3 回归测试

- 提示词版本切换
- provider fallback
- 并发高峰下的租约释放

## 18. 首期实施边界

虽然本设计是完整平台方案，但首个实施版本应保持以下边界：

- 支持多 provider 和多 key
- 支持固定数量平台连接器
- 支持完整控制台
- 支持关键节点对话确认
- 不实现登录和权限
- 不实现多租户
- 不实现开放 API

## 19. 风险与缓解

### 19.1 长篇一致性漂移

缓解：

- 故事状态中心
- 章节摘要
- 审核 agent
- 关键节点人工确认

### 19.2 provider 或 key 耗尽

缓解：

- 容量租约
- fallback
- 预算控制

### 19.3 过度并行导致剧情冲突

缓解：

- 单小说正文单线程
- 关键节点阻塞机制

### 19.4 提示词变更引发质量波动

缓解：

- 提示词版本化
- 测试运行
- 回滚

## 20. 推荐实现顺序

1. 领域模型与数据库 schema
2. LLM 调度器
3. `superpower` 核心流程
4. 大纲、分部、章节、审核、改写 agent
5. 人工确认页与剧情确认助手
6. 发布中心与平台连接器
7. 控制台剩余页面

## 21. 结论

这个系统的核心竞争力不在于 agent 数量，而在于以下五个中枢是否稳定：

- 故事状态中心
- `superpower` 工作流
- LLM 容量调度器
- 关键节点决策会话
- 发布中心

只要这五个边界清晰，模型可以替换，提示词可以迭代，平台也可以继续扩展，而整体生产链路不会失控。
