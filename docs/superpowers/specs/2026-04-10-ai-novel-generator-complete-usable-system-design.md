# AI 小说生成器完整可用系统设计

## 1. 目标

本设计面向单人内部使用场景，不引入鉴权与多用户协作复杂度，目标是把当前项目补完为一个“可以完整使用”的小说生产系统。

完成后的系统应满足以下标准：

- 用户可以从已创建项目出发，自动推进小说生产主链
- 系统默认持续运行，直到遇到必须人工确认的关键节点或明确失败
- 人工确认节点会提供 2-3 个推荐选项，并允许用户补充自然语言说明
- 单章审核不要求人工确认，审核与有限次自动改写应默认自动推进
- 系统在中断、失败、人工暂停后能够从明确恢复点继续
- 已批准章节与阶段性成果可以由用户手动导出为文档

## 2. 适用范围

### 2.1 本设计包含

- 自动生产主链：总纲、卷规划、章节计划、章节草稿、自动审核与自动改写
- 统一人工确认模型：总纲确认、卷规划确认、阻塞决策确认、续接确认
- 断点续接与恢复动作执行
- 手动导出能力
- Worker、Workflow、Storage、Web 控制台之间的完整闭环

### 2.2 本设计不包含

- 用户鉴权、组织协作、权限模型
- 外部平台自动上传发布
- 实时多人编辑
- 云端部署、运营、监控平台化建设

## 3. 产品定义

系统是一个“自动生成主链 + 少量人工 gate + 推荐选项辅助决策 + 手动导出”的内部工作台。

默认运行模式：

1. 用户启动或继续项目
2. 系统自动沿工作流推进
3. 在关键节点暂停并创建人工确认会话
4. 用户选择推荐项或补充说明
5. 系统根据确认结果继续推进
6. 用户在合适时机手动导出内容

这意味着系统不是“每一步都人工驱动”，也不是“完全无人值守”，而是一个以自动推进为主、关键处由用户拍板的协作式小说生成系统。

## 4. 关键人工确认节点

系统只在以下节点强制等待人工确认：

- `outline_confirmation`
- `volume_confirmation`
- `blocked_decision`
- `resume_confirmation`

明确不设为强制人工确认的节点：

- 单章审核结果
- 单章自动改写结果
- 导出前审批

导出行为由用户主动触发，但不要求额外审批 gate。

## 5. 总体架构

系统由 5 个层次构成：

### 5.1 生产执行层

负责 outline、volume、chapter、review/rewrite 的自动执行。

该层特点：

- 默认持续推进
- 负责真实模型调用与持久化
- 不直接处理人工确认 UI

### 5.2 人工 Gate 层

负责所有需要用户拍板的节点。

统一职责：

- 基于当前上下文生成 2-3 个推荐选项
- 展示推荐理由和潜在影响
- 接收用户选择与自然语言补充
- 形成可恢复的确认结果

### 5.3 恢复续接层

负责系统在失败、中断、人工暂停后的重新启动。

统一职责：

- 明确系统停在何处
- 提供推荐续接方案
- 允许用户确认后继续执行

### 5.4 导出层

负责将已批准内容组织为导出文档。

支持目标：

- 单章节导出
- 单卷导出
- 已批准整书导出

### 5.5 控制台层

Web 端负责展示当前进度、待确认项、推荐选项、恢复点和导出入口。

## 6. 核心对象模型

系统围绕以下核心对象工作：

### 6.1 WorkflowRun

表示一次自动推进任务。

职责：

- 记录当前运行的 flow
- 记录步骤执行状态
- 标识成功、失败、等待人工确认

### 6.2 StoryState

表示小说当前的正式持久化状态。

至少包含：

- 已确认总纲
- 已确认卷规划
- 已确认章节摘要
- 当前章节位置
- 事实与伏笔状态

### 6.3 ChapterArtifact

表示章节相关的阶段性产物。

至少包括：

- ChapterPlan
- ChapterDraft
- ReviewOutcome
- Approved 章节摘要

### 6.4 HumanGateSession

表示统一的人机确认节点。

这是本设计的关键抽象，用于替代当前只面向阻塞审核的狭义 decision session。

每个 gate session 至少包含：

- gate 类型
- 当前上下文快照
- 2-3 个推荐选项
- 每个选项的推荐理由
- 用户最终选择
- 用户补充说明
- 后续恢复动作

## 7. 核心状态流转

推荐的顶层状态流如下：

1. 项目创建
2. 自动生成总纲
3. 等待总纲确认
4. 自动生成卷规划
5. 等待卷规划确认
6. 自动生成章节计划与章节草稿
7. 自动审核与自动改写
8. 若章节批准，则推进到下一章
9. 若自动处理失败或冲突无法解决，则等待阻塞决策确认
10. 若系统中断、人工暂停或异常恢复，则等待续接确认
11. 已批准内容可随时手动导出

状态流的关键原则：

- 单章审核不阻塞用户
- 总纲与卷规划一旦确认，成为后续生成基线
- 阻塞决策优先做局部修正，不默认全局重置
- 任意人工 gate 结束后，系统都应拥有明确的下一步恢复动作

## 8. 推荐选项模型

每个 gate 默认生成 2-3 个推荐选项，推荐结构统一：

- `recommended`：系统最推荐方案
- `alternative`：有明确取舍差异的替代方案
- `custom_seed`：允许用户补充自然语言的半结构方案

每个选项必须展示：

- 方案标题
- 推荐等级
- 适用原因
- 预计影响

用户交互支持：

- 直接采用某个推荐项
- 采用某项并补充说明
- 使用自己的描述覆盖推荐项

## 9. 业务链设计

### 9.1 总纲链

流程：

1. 读取项目与 prompt
2. 执行总纲生成
3. 写入 StoryState
4. 创建 `outline_confirmation` gate

系统推荐选项应聚焦：

- 直接采用当前总纲
- 保守调整版
- 可补充说明的自定义版

确认通过后进入卷规划链。

### 9.2 卷规划链

流程：

1. 基于已确认总纲生成卷规划
2. 写入 StoryState
3. 创建 `volume_confirmation` gate

系统推荐选项应聚焦不同结构策略：

- 稳健展开
- 节奏偏快
- 铺垫偏长

确认通过后进入章节自动生产链。

### 9.3 章节自动生产链

流程：

1. 自动生成 ChapterPlan
2. 自动生成 ChapterDraft
3. 自动执行 review/rewrite
4. 审核通过后回写摘要并推进下一章
5. 审核在自动改写限制内失败则自动重试
6. 超出限制则进入 `blocked_decision`

章节链默认全自动推进，不要求人工逐章审核。

### 9.4 阻塞决策链

当自动生产无法继续时，系统不只是停下，还必须给出恢复建议。

推荐项示例方向：

- 保留当前方向，小范围重写当前章
- 调整后续若干章计划
- 回退特定设定并继续

用户确认后，系统把该选择转化为明确恢复动作，再进入恢复续接层。

### 9.5 续接链

续接不是简单“继续运行”，而是要先明确恢复点。

续接确认页必须说明：

- 系统上次停在什么步骤
- 哪一章或哪一段规划受影响
- 推荐从哪里恢复
- 是否需要先修改上下文

用户确认后再创建新的 WorkflowRun 继续执行。

### 9.6 导出链

导出不参与自动审批流，只面向已批准内容。

支持：

- 当前章节导出
- 指定卷导出
- 已批准整书导出

格式支持：

- `plain_text`
- `markdown`
- `bundle`

## 10. Workflow 与 Gate 的编排模型

推荐把 gate 视为 workflow 的正式暂停状态，而不是一条旁支逻辑。

推荐执行语义：

1. WorkflowRun 启动
2. 自动步骤持续执行
3. 一旦进入需要人工确认的节点：
   - 当前 WorkflowRun 结束为 `waiting_for_human_gate`
   - 创建 HumanGateSession
4. 用户完成确认后：
   - 创建新的 WorkflowRun
   - 从恢复点继续

这样做的好处：

- 断点续接边界清晰
- 审计路径清晰
- 每次人工决策后都能单独形成新的恢复链路

## 11. Web 控制台设计

建议保留 3 个主工作面：

### 11.1 项目工作台

展示：

- 当前阶段
- 已确认基线
- 章节进度
- 最近失败或阻塞信息
- 快速继续入口

### 11.2 待确认队列

展示所有待处理 gate session。

每个卡片展示：

- gate 类型
- 当前上下文摘要
- 系统推荐项
- 是否存在恢复风险

### 11.3 导出中心

负责选择导出范围和格式，并查看导出结果。

## 12. Gate 交互页面设计

所有 gate 使用统一页面模型：

1. 顶部说明区
   - 系统为什么停下
   - 当前上下文摘要
   - 继续后将发生什么
2. 中部推荐区
   - 2-3 个推荐选项卡
   - 明确标识系统推荐项
3. 自由输入区
   - 用户补充自然语言说明
4. 操作区
   - 采用推荐方案
   - 采用并附加说明
   - 使用我的说明生成继续方案

交互原则：

- 默认突出一个推荐方案
- 其他方案作为对照，而不是让用户每次重新完整推理

## 13. 错误处理

错误分为 3 类：

### 13.1 可自动重试

例如：

- 模型调用失败
- 容量租约暂时失败
- 轻微结构化输出错误

### 13.2 需要人工 gate

例如：

- 总纲质量不稳
- 卷规划方向冲突
- 章节自动改写耗尽
- 恢复路径存在多个合理方案

### 13.3 硬失败

例如：

- 关键持久化失败
- 状态机非法跳转
- 必需上下文缺失

系统要求：

- 任何失败都必须留下清晰恢复点
- 不允许出现无法解释的停机状态

## 14. 测试策略

建议测试分 4 层：

### 14.1 单元测试

覆盖：

- 推荐选项生成
- gate 状态跳转
- 恢复动作解释

### 14.2 Workflow 测试

覆盖：

- 正常自动推进
- 进入人工 gate
- 恢复后继续
- 失败路径

### 14.3 仓储与集成测试

覆盖：

- StoryState
- HumanGateSession
- RecoveryTask
- ExportArtifact

### 14.4 端到端 Smoke

至少验证一条完整主线：

- 创建项目
- 生成总纲
- 人工确认总纲
- 生成卷规划
- 人工确认卷规划
- 自动生成章节
- 遇阻塞后确认恢复方案
- 继续生成
- 手动导出

## 15. 完成标准

系统可视为“完整可用”，至少要满足：

- 单人无需鉴权即可完整使用
- 系统能自动推进到下一个人工 gate
- 用户能在所有关键 gate 看到推荐选项并继续
- 中断后能恢复
- 已批准内容可手动导出
- 主链 smoke 与完整回归测试通过

## 16. 实施拆分

推荐拆为 4 个后续实现 spec：

### 16.1 子项目一：统一 HumanGate 模型 + 总纲/卷规划确认

目标：

- 建立统一 gate session
- 接入 outline 与 volume 确认

### 16.2 子项目二：章节自动推进主链 + review/rewrite + 自动续章

目标：

- 把章节生产链做成连续自动推进
- 审核通过后自动进入下一章

### 16.3 子项目三：blocked decision + resume confirmation + recovery 执行

目标：

- 完成阻塞决策推荐
- 完成恢复动作与续接

### 16.4 子项目四：导出工作台 + 整书/分卷/章节导出闭环

目标：

- 补齐导出交互和可交付文档能力

这样拆分后，每一阶段都能形成一个更完整、可运行的系统增量，而不是只堆积局部能力。
